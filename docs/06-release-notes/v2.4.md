## Mobile

### Asset Lookup (Sprint M4-BE1)

`GET /api/mobile/assets/lookup?code={assetCode}` resolves a stable asset business code (`AST-XXXXXXXX`) to a scoped operational context. Every asset has a `code` field independent of the database primary key (migration `V28__asset_business_code.sql`).

### Asset Context

The lookup response is a single-screen bundle (`AssetContextResponse`):

| Field | Description |
|-------|-------------|
| `asset` | Summary (id, code, name, category, department, location, status) |
| `lastInspection` | Most recent completed inspection, or `null` |
| `lastMaintenance` | Most recent completed maintenance activity, or `null` |
| `preventivePlan` | Active preventive plan summary, or `null` |
| `documents` | Visible asset-owned operational documents (always an array) |
| `openIssues` | Unresolved issues on the asset |
| `activeInspections` | Inspections with status `ASSIGNED` |
| `activeWorkOrders` | Work orders with status `CREATED` or `ASSIGNED` |
| `allowedActions` | Backend-generated capability flags |

Enrichment sprints: M4-BE3 (last inspection/maintenance/preventive plan), M4-BE4/M4-BE4.1 (documents and field-employee visibility fix).

### QR foundation (Sprint M4-BE2)

`GET /api/assets/{assetId}/qr` returns a PNG QR (512Ă—512, high error correction) encoding the asset business code only. No database ids, URLs, or JSON are embedded.

**End-to-end flow (when Android scanning ships):**

```text

        â†“
```

### Documents

Asset-owned operational documents appear in the `documents` array with metadata and `downloadUrl`. Download reuses the existing operational document endpoint with mobile-aligned authorization for asset-owned documents.

### Allowed Actions

`allowedActions` is generated entirely by the backend (`MobileAuthorizationService`). Android must not infer permissions from role alone. Flags include `canViewAsset`, `canViewInspections`, `canViewIssues`, `canViewWorkOrders`, `canCreateInspection`, and `canCreateIssue`.

## Reporting

### Export formats

Read-only exports for five operational domains:

| Format | Path pattern |
|--------|--------------|
| CSV | `/api/reporting/exports/*.csv` |
| XLSX | `/api/reporting/exports/*.xlsx` |
| PDF | `/api/reporting/exports/*.pdf` |

Domains: Assets, Inspections, Issues, Work Orders, Preventive Execution Candidates.

### Export menu (Security-2.1 / frontend)

List pages use a single **Export** dropdown (`ExportReportingMenu`) offering CSV, XLSX, and PDF. The menu includes **from** and **to** date inputs defaulting to the last 30 days (inclusive). Client-side validation mirrors server rules before download.

Authorized roles: Administrator, Manager, Operational Coordinator.

### Required export date range (Security-2)

Every export request must include `from` and `to` query parameters (epoch millis). The server rejects requests with either parameter omitted.

### 365-day validation

The export window cannot exceed **365 inclusive calendar days** (UTC). `to` must not precede `from`.

| Condition | HTTP status |
|-----------|-------------|
| `from` or `to` omitted | `400` |
| `to` before `from` | `400` |
| Window > 365 days | `400` |

### Formula injection protection (Security-1)

CSV and XLSX writers prefix cell values starting with `=`, `+`, `-`, `@`, or TAB with a single apostrophe. PDF exports are unchanged. Implementation: `ExportCellFormatter.sanitizeSpreadsheetText()`.

## Security

### Content Security Policy (DT-2A)

Production CSP is enforced on the **frontend nginx** container. JWT Bearer authentication and `localStorage` token storage are unchanged (BDR-003).

Current directives: `default-src 'self'`, `script-src 'self'`, `style-src 'self'`, `connect-src 'self' https: http://localhost:4000`, `object-src 'none'`, `frame-ancestors 'none'`.

**Limitations:** `upgrade-insecure-requests` is not included â€” HTTP deployments remain supported. API CSP on the Spring Boot backend is intentionally deferred.

**Future tightening:** Reintroduce `upgrade-insecure-requests` when HTTPS is mandatory for all deployments; consider API response CSP when the SPA and API share a single origin.

### Disabled user immediate revocation (Security-3)

After JWT signature validation, `JwtAuthenticationFilter` checks `UserAccountStatusService.isEnabled(userId)`:

| Control | Setting |
|---------|---------|
| Cache | Caffeine, per-user enabled flag |
| TTL | 30 seconds |
| Max size | 10,000 entries |
| Eviction | On deactivate, reactivate, and account activation |

Disabled users receive HTTP `401` on the next request after status change (immediate when cache is evicted).

### JWT account validation cache

The 30-second cache balances database load against revocation latency. Eviction on status change provides immediate revocation in practice; worst-case latency without eviction is 30 seconds.

### Spreadsheet formula protection

See Reporting section above and [security.md](../05-deployment/security.md#spreadsheet-formula-injection-protection).

### Authorization architecture guard

`AuthorizationArchitectureTest` (ArchUnit-style scan) verifies every `@RestController` depends on an `*AuthorizationService` directly or through an application service. Controllers without an authorization path must be explicitly allowlisted with documented justification.
